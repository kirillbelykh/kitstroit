# Эксплуатация

## Среды

- `local` — разработка через Docker Compose.
- `production` — один VPS, отдельные volumes для PostgreSQL, медиа и сертификатов.

Staging добавляется только когда релизы станут достаточно частыми, чтобы локальной проверки и production health check было недостаточно.

## Секреты

- `.env.example` содержит только имена переменных и безопасные примеры.
- Production `.env` создаётся на сервере с правами `600`.
- Токен Telegram, пароль администратора, ключ сессии и пароль БД не попадают в Git, CI-логи и Docker image.
- Доступы, которые когда-либо передавались в открытом тексте, заменяются до первого деплоя.
- Ротация секрета фиксируется датой, но не значением.

## Деплой

1. Проверить чистый Git tag и успешные тесты.
2. Создать резервную копию БД и медиа.
3. Собрать образы с тегом релиза.
4. Выполнить `alembic upgrade head` отдельной одноразовой командой.
5. Запустить Compose и дождаться health checks.
6. Проверить главную, форму без отправки, вход в админку, медиа и `/api/health`.
7. Отправить одну тестовую заявку и подтвердить запись в БД и доставку в Telegram.

На production VPS используется `compose.host.yaml`: существующая сетевая схема не допускает создание Docker bridge без вмешательства в системные firewall-цепочки. PostgreSQL, backend и frontend при этом слушают только `127.0.0.1`; публичные порты `80/443` принадлежат Caddy. Образы собираются командой `docker build --network host`, после чего стек запускается через `make prod-up`. Сетевые правила и сторонние контейнеры при деплое сайта не изменяются.

Не использовать `latest` как единственный тег production-образа.

## Резервные копии

Ежедневно:

- `pg_dump` в custom format;
- архив каталога медиа;
- копия конфигурации reverse proxy без секретов.

Хранение: 7 ежедневных и 4 еженедельных копии. Минимум одна копия должна находиться вне VPS. Раз в месяц выполняется тестовое восстановление во временную БД и сверка количества заявок, проектов и медиа.

## Откат

1. Остановить дальнейший деплой.
2. Зафиксировать ошибку и текущий release tag.
3. Вернуть предыдущие образы.
4. Если миграция несовместима — восстановить предрелизный dump; не выполнять ручной downgrade на production без проверки.
5. Проверить health endpoint, публичную страницу, вход и последнюю заявку.

Медиа не удаляются при откате приложения. Замена файлов выполняется атомарно, поэтому предыдущий файл сохраняется до успешной записи нового.

## Мониторинг

- Docker health checks для backend и PostgreSQL.
- Проверка доступности главной и `/api/health` раз в минуту внешним монитором.
- Алерты: недоступность сайта, ошибки 5xx, заполнение диска выше 80%, неуспешный backup.
- Telegram-сбой не считается потерей заявки: в админке видны `pending` и `failed` уведомления.
- Логи ротируются и не содержат секретов или полных персональных данных.

## Аварийный минимум

- При нехватке диска сначала остановить загрузку новых медиа, не удалять БД и backups вслепую.
- При компрометации немедленно закрыть доступ, заменить все секреты, сохранить логи и восстановиться из проверенной копии.
- При проблемах Telegram оставить приём заявок включённым и обрабатывать сохранённые записи из админки.
